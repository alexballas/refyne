Client-Side Decorations Comparison: Refyne vs go-gl/glfw v3.4/glfw/v0.1.0-pre.1

Updated: 2026-07-02 22:18 EEST Local repo: 1bf007f28541 go-gl tag: 9c147ed2fc8c GLFW C source: 7b6aead9fb88

Executive Summary

The local Refyne tree vendors github.com/go-gl/glfw/v3.4/glfw v0.1.0-pre.1 under internal/glfw, then adds a Wayland client-side decoration layer that is not present in the upstream go-gl tag.

The vendored GLFW-side implementation is concentrated in wl_refyne.c, wl_window.c, wl_platform.h, wl_init.c, generated Wayland protocol bindings, and a Go wrapper file. It adds compositor move/resize/menu calls, decoration-mode detection, custom cached shadow subsurfaces that double as resize handles, cursor-shape-v1 support, xdg-toplevel-icon support, and EGL/Wayland commit timing guards for stable interactive resize.

There is also a Refyne driver/UI layer outside internal/glfw: it installs the in-canvas title bar, reserves canvas space for it, handles edge resize and title-bar menu gestures, updates rounded/square corners, pushes icons, and avoids resize fights while the Wayland compositor is driving an interactive resize.

The full vendored GLFW source diff against the requested go-gl tag is 32 files, 3,838 insertions, and 188 deletions. Not all of that is CSD: FileTransfer portal support, X11 drop handling, module-path rewrites, and formatting/import cleanups are adjacent or unrelated and are separated below.

Comparison Method

Baseline source was checked out from the requested release tag and compared with the local vendored module root:

git clone --depth 1 --branch v3.4/glfw/v0.1.0-pre.1 https://github.com/go-gl/glfw.git /tmp/refyne-csd-compare/go-gl-glfw
git diff --no-index /tmp/refyne-csd-compare/go-gl-glfw/v3.4/glfw internal/glfw

The go-gl release tag resolves to 9c147ed2fc8c8daefcbb3225e10a059af407e53c. The vendored GLFW C revision is 7b6aead9fb88b3623e3b3725ebb42670cbe4c579, matching the dereferenced GLFW 3.4 tag and internal/glfw/GLFW_C_REVISION.txt.

Scope Note

This report treats "client-side decorations" as the Wayland decoration behavior used when the compositor either grants client-side xdg-decoration mode or does not provide server-side decorations. That includes the visible custom shadow/resize handles, fallback decoration pointer behavior, cursor selection for those handles, and the resize/commit synchronization needed to keep those surfaces stable during compositor-driven resize.

Implementation Flow

  1. At build time, wl_refyne.c is included after upstream wl_window.c, allowing Refyne helpers to call GLFW internal Wayland functions and use internal structs.
  2. At Wayland registry setup, Refyne binds wp_cursor_shape_manager_v1 and xdg_toplevel_icon_manager_v1 if the compositor advertises them.
  3. The Refyne driver creates the window with Wayland app-id and transparent-framebuffer hints, then waits for decoration mode. Server-side decorations are left alone; otherwise Refyne disables GLFW fallback decorations and installs its own title bar in the canvas.
  4. When Refyne requests custom decorations, glfwRefyneSetWindowShadow marks shadow state as requested and calls _glfwRefyneUpdateWindowShadow.
  5. The shadow updater creates a single wl_shm atlas, eight subsurfaces around the content surface, viewports into that atlas, and xdg window geometry that excludes the shadows.
  6. The canvas reserves title-bar height, offsets content/popups, paints rounded transparent corners plus an opaque body, and applies current theme colors to the custom chrome.
  7. Pointer enter/motion/button handling distinguishes content, fallback decoration surfaces, and Refyne shadow slices. Refyne's driver also intercepts canvas-edge presses and title-bar right-clicks before normal widget dispatch. Shadow slices and canvas-edge hit zones map to xdg resize edges and issue compositor resize grabs.
  8. During configure/resize, sizeCommitPending prevents bare commits from applying new viewport or geometry state against an old content buffer. The matching eglSwapBuffers clears the flag, while InteractiveResizingWayland lets Refyne avoid fighting compositor-driven resize.
  9. Cursor selection prefers compositor-rendered cursor-shape-v1 resize cursors; if unsupported, it falls back to libwayland-cursor theme buffers and additional XDG/X11 cursor aliases. The Refyne driver keeps the frame resize cursor active while the pointer is in its edge hit zone.
  10. Cleanup invalidates pointer focus before destroying CSD/fallback/content surfaces, destroys shadow subsurfaces and buffers, and releases protocol objects during Wayland termination.

Adjacent Non-CSD Changes

These are present in the comparison but are not part of the client-side decoration implementation:

  • file_transfer_portal.[ch], c_glfw.go, internal.h, x11_*.c/.h, and parts of wl_window.c/wl_init.c implement optional FileTransfer portal drops.
  • build_cgo_hack.go, glfw/deps/dummy.go, and the removed nested go.mod are vendoring/module-path changes.
  • native_linbsd_x11.go, not_wayland.go, vulkan.go, and window.go contain build-tag, formatting, or wrapper cleanups unrelated to CSD behavior.
  • Several Wayland hunks are post-GLFW-3.4 robustness backports that support the CSD paths but are not unique to decorations: key repeat hardening, data-offer cleanup, pointer-constraints guards, callback cleanup, and delayed module unload.
